Background

What is the Problem?

Machine Learning Systems consist of chained models. Dataset shifts force periodic retraining — but retraining can break things in unexpected ways.

The Catch: Retraining upstream model u might improve u individually, yet degrade the overall system output
Core Problem
Entangled Enhancement

An upstream retraining that individually improves a component but paradoxically degrades overall system performance due to tight component dependencies.

  • Models learn to operate on specific output characteristics of their upstream neighbor
  • After retraining, output distribution shifts — downstream model sees unfamiliar inputs
  • System-level performance drops even though component-level metric improved
  • Very difficult to diagnose: it looks like the update was successful

kij(t) = P{Xn+1=j, Tn+1−Tn≤t | Xn=i}

Kernel function: probability of transitioning from state i to j within time t

πi = vihi / Σj vjhj

Steady-state probability with general sojourn time distributions (Weibull)

Why SMP over CTMC?
  • CTMC forces exponential sojourn time — unrealistic for real retraining processes
  • SMP allows any distribution (Weibull used: shape β captures increasing/decreasing hazard rates)
  • Shape parameter β affects availability predictions — CTMC misses this entirely
  • First-ever application of SMP to model MLS with retraining policies

Maintenance Policies

Progressive vs Conservative Retraining

Two fundamental strategies for managing MLS components under dataset shift.

Progressive Policy (πP)

Retrain models proactively whenever any component degrades — even if system output is still acceptable.

  • At state (0,1): retrain upstream u immediately
  • At state (1,0): retrain downstream d
  • At state (0,0): retrain both u and d
  • Higher service availability in most conditions
  • Risks triggering entangled enhancement at (0,1)
Conservative Policy (πC)

Retrain only when system output is degraded — never retrain proactively when system is still serving.

  • At state (0,1): do nothing (system still available)
  • At state (1,0): retrain downstream d
  • At state (0,0): retrain both components
  • Higher service continuity (fewer interruptions)
  • Lower availability — misses recovery opportunities

Key Finding

The Availability–Continuity Trade-off

The most important insight: these two metrics are in fundamental tension, and policy choice determines where you land.

Service Availability (A)

Long-run probability the system meets user satisfaction — the downstream model is above threshold.

Average time between transitions from available to unavailable states. Higher = more stable service.

Sensitivity Analysis Results
  • η₂ (downstream failure rate) — most critical parameter for both metrics
  • η₃, η₄ (retraining rates) — increasing them boosts availability but hurts continuity
  • c₁ close to 0 → progressive becomes worse than conservative in both metrics
  • Shape parameter β matters — same MTTR but different β gives different availability
Strategy Guide:
  • 🏥 Patient care → Conservative (safety-critical real-time accuracy)
  • 🏭 Predictive maintenance → Progressive (long-term avg accuracy)
  • ⚠️ If c₁ < 0.1 → Always Conservative
  • ⚠️ If η₄ < 0.064 → Always Conservative
Master's Thesis Contribution

Dynamic Retraining Policy

Goes beyond the two static policies by introducing a monitoring mechanism that selectively accepts or rejects retraining attempts.

Mechanism
1

Upstream failure detected

Dataset shift degrades model u below threshold

2

Retraining attempted

New model u' is trained

3

Evaluation metric assessed

WACR or Regret Ratio evaluated against threshold

4

Accept or Reject

Pass → Progressive mode · Fail → Conservative mode

WACR Metric
WACR = Σwᵢ·1(yᵢ=ŷᵢ) / Σwᵢ

Weights critical data regions (e.g., 20m+ objects) more heavily when evaluating retrained model

Regret Ratio
RR = New Errors / Fixed Errors

Penalizes retraining that fixes some things while breaking previously correct predictions

Utility Maximization (Theorem)

For any continuous utility function u(A,C), dynamic policy always achieves a global maximum utility — proved via extreme value theorem on the bounded ROC space.

My Research Proposal

Proactive MLOps Control Framework

Extends static policy analysis into a scalable, predictive, adaptive MLOps system for multi-component MLS.

Gap 1: Scalability

CTMC/SMP face 2ᴺ state explosion. Need a computationally tractable approach for N>2 components.

Proposed: Simulation + ML predictor
Gap 2: No Prediction

Prior work evaluates policies post-hoc. No tool to assess entanglement risk before retraining.

Proposed: Entanglement Forecaster
Gap 3: Static Policies

Fixed progressive/conservative strategies cannot adapt dynamically to real-time system context.

Proposed: Intelligent Agent

Phase 1 — Entanglement Forecaster

  • SimPy-based MLS simulation with multi-component architectures
  • Feature engineering: output distribution shift, historical entanglement patterns, inter-model coupling
  • Gradient Boosting / Random Forest trained on simulated data
  • Outputs a risk score per retraining candidate

Phase 2 — Smart Navigator Agent

  • Low Risk: Standard isolated retraining
  • Moderate Risk: Retrain + intensified downstream monitoring
  • High Risk Option A: Pause → expert review → feedback loop
  • High Risk Option B: Coordinated upstream + downstream retrain
  • High Risk Option C: Constrained/regularized retraining

New Failure Mode 1

Cascade Degradation

Retraining C1 degrades C2, which in turn degrades C3. Second-order failure propagates silently through the chain.

New Failure Mode 2

Partial Recovery Ambiguity

C1 and C3 improve but C2 degrades. System state is undefined under binary thresholds — needs weighted scoring.

New Failure Mode 3

Policy Complexity Explosion

2³-1 = 7 non-trivial retraining subsets per decision. Unconstrained policies: 8⁸ ≈ 1.68×10⁷

Why nuScenes? Strict sequential dependency · Independent retrainability · Natural domain shift (Boston→Singapore) · Safety-critical stakes · Continues KITTI thread from base paper

4 Unexplored Policy Families
πDF

Downstream-First

At (0,0): retrain only d first. Avoids Type 2 entanglement risk from concurrent upstream retraining.

πJ

Joint Retraining

Retrain {u,d} whenever anything degrades. Maximum aggression — doubles cost but maximizes recovery rate.

πUA

Upstream-Avoidant

Never retrain u alone. Eliminates both Type 1 and Type 2 entanglement by construction.

πTC

Threshold-Conditional

Tunable γ: πTC(0,1) = {u} if c₁≥γ, else ∅. Progressive (γ=0) and Conservative (γ=1) are special cases — the two policies are endpoints of a spectrum.

Meeting 2 — Evaluation Framework

Three-Tier Evaluation for Planning (C3)

Planning is fundamentally harder to evaluate than perception or prediction — three challenges require a dedicated framework.

Why Planning is Hard
  • No unique ground truth — multiple valid plans exist; displacement metrics penalize equally-valid alternatives
  • Silent degradation — on simple straight roads, even a broken planner outputs "go straight"
  • Upstream masking — C3 might be fine, but appears degraded because C1 missed a pedestrian
Error Propagation Ratio:
ρᵢ→ⱼ = (ΔMⱼ/Mⱼ) / (ΔMᵢ/Mᵢ)

If C1 mAP drops 5% and C3 L2 error rises 20% → ρ = 4 (strong coupling). Directly informs SMP coverage factors.
T1

Tier 1 — Plan Quality

L2 displacement error (primary), collision rate, drivable area compliance, comfort (jerk/lateral acceleration)

T2

Tier 2 — Component Isolation

Evaluate C3 with ground-truth detections & predictions as input (bypass C1, C2). Compare to pipeline output to diagnose fault source.

GT-in Good + Pipeline Bad → upstream fault
GT-in Bad + Pipeline Bad → C3 itself degraded
T3

Tier 3 — Coupling Metrics

Error propagation ratio ρᵢ→ⱼ quantifies how strongly upstream errors amplify through the chain. Used to calibrate SMP coverage factors from real data.

This framework produces clean binary s₁, s₂, s₃ signals for the 8-state SMP model — resolving the core measurement challenge for three-component analysis.
