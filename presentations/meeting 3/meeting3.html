Likely Questions & Answers
Why PDM-Closed and not an end-to-end model like UniAD?
End-to-end models like UniAD integrate perception, prediction, and planning into a single network. We cannot independently retrain one component and isolate its effect. For our research we need three separately retrainable, separately evaluable components — PDM-Closed gives us that modularity.
How do you know the Singapore split is a strong enough domain shift?
Boston and Singapore differ in road geometry, traffic density, driving side, and agent behavior. The nuScenes community uses this split specifically as a cross-city generalization benchmark. If anything, it may be conservative — real-world deployment shifts can be larger.
What if E1 shows no cascade degradation?
That is also a valid result. It would mean the two-component entanglement does not extend to three components in this specific pipeline, which would itself be an important finding. We would then investigate whether the intermediate C2 absorbs or amplifies the upstream shift, and design follow-up experiments accordingly.
Why MTP as the prediction baseline rather than AgentFormer?
MTP is lightweight and fast to train, which is practical for an initial experiment. If the cascade effect is confirmed, we can re-run with a stronger prediction model like AgentFormer to test whether prediction model quality changes the magnitude of the cascade.

Section II

What Is the Planning Module?

C3 is the final component. Its output is a physical trajectory — a real path the car will follow.

What C3 Receives
  • From C1 (Perception): list of detected objects with current positions and sizes
  • From C2 (Prediction): forecast of where each object will be in the next 2–6 seconds
What C3 Produces

A trajectory — a sequence of positions and velocities the ego-vehicle will follow over the next few seconds.

ANSWER: given where everything is and where it is going, what path should this vehicle take?

The Core Vulnerability: C3 never sees raw sensor data. It only sees what C1 and C2 give it. If either is wrong, C3 builds a plan on a false picture of the world — optimal for the world it thinks it is in, but dangerous for the world it actually is in.
Why C3 Is the Right Focus
  • All upstream errors eventually arrive here
  • Failure is not just a bad metric — it can be a collision
  • C3 cannot distinguish "world is safe" from "world looks safe"
Section III

Which Planning Model to Use

We select PDM-Closed as C3. Three reasons.

Reason 1

Modularity

PDM-Closed is a standalone module — takes predicted trajectories as input, outputs an ego-trajectory. Does not jointly train with C1 or C2. Essential for independent evaluation.

Reason 2

nuScenes Compatible

Top-performing model on the official nuScenes planning benchmark. Results are directly comparable to community baselines — gives the research credibility.

Reason 3

Interpretable

Uses a scoring function over candidate trajectories. When output degrades, the cause is traceable to changed inputs — not to C3 itself.

Primary Metric — L2 Displacement Error:
Distance between planned trajectory and human driver's trajectory at 1s, 2s, and 3s horizons. Measures planning accuracy.
Safety Metric — Collision Rate:
Percentage of scenarios where the planned path intersects another agent. Measures real-world danger.

The Cascade Mechanism: When C1 is retrained, its output distribution changes. C2 now receives inputs it was NOT trained on. C2's predictions shift. C3 receives corrupted predictions. C3's plan degrades — and C3 was never retrained.
Why This Is Hard to Detect
  • C1's own mAP metric may have improved
  • C2's isolated performance is unchanged
  • C3's isolated performance is unchanged
  • Only the end-to-end pipeline gets worse
  • Looks like a successful update. System gets worse.
Section V

What Happens With Faulty Upstream Data

Three concrete failure modes for C3 after upstream retraining.

Failure Mode 1

Missing Detections

C1 misses objects it previously detected — e.g., pedestrians at long range. C2 never predicts their movement. C3 plans as if they don't exist.

Most dangerous. Also silent — C1's aggregate mAP may improve while long-range recall drops.
Failure Mode 2

Shifted Distributions

C1 still detects all objects, but bounding box sizes, confidence scores, or center positions shift slightly. C2 receives inputs outside its training distribution.

C2's isolated minADE is unchanged. But pipeline minADE worsens. Tier 2 framework detects this directly.
Failure Mode 3

Compounded Error

Both effects together: fewer agents detected AND predictions less reliable for those that are detected. C3 faces maximum uncertainty.

L2 error increases, collision rate rises. This is what Experiment E1 is designed to observe.
Key insight from Meeting 2: Tier 2 of the evaluation framework — comparing C3 with ground-truth inputs vs. live pipeline inputs — is specifically designed to distinguish these three failure modes from genuine C3 degradation.
Section VI — Proposed Experiment

Initial Experiment Setup

One small, targeted experiment to confirm cascade degradation before any modeling work begins.

Step 1 — Build the Baseline

Train all three components independently on the nuScenes Boston split.

  • C1 CenterPoint — 3D detection, metric: mAP
  • C2 MTP Baseline — prediction, metric: minADE
  • C3 PDM-Closed — planning, metric: L2 + collision

Record baseline performance of each component and full end-to-end pipeline on the validation set.

Step 2 — Inject Domain Shift

Evaluate the trained system on the nuScenes Singapore split. Different road geometry, opposite-side driving, different agent density.

This is a natural, reproducible shift — not synthetic. All three components are affected differently.

Step 3 — The Critical Test (E1)

Retrain C1 only using Singapore data. Do not retrain C2 or C3.

If C3 (pipeline) degrades while C3 (isolated) stays the same — cascade entanglement is confirmed.
  • C1 mAP should improve (retrained for Singapore)
  • C2 isolated: unchanged
  • C2 pipeline: may worsen
  • C3 isolated: unchanged
  • C3 pipeline: may worsen ← the key observation
Section VI — Measurement Table

What to Measure

Five measurements. The gap between isolated and pipeline tells us everything.

Component Evaluation Mode Metric Expected After C1 Retrain
C1 Live pipeline mAP on Singapore ↑ Improves — this is the point of retraining
C2 (isolated) Ground-truth detections as input minADE Unchanged — C2 not retrained
C2 (pipeline) Live C1 detections as input minADE ↓ May worsen — input distribution shifted
C3 (isolated) Ground-truth predictions as input L2 error Unchanged — C3 not retrained
C3 (pipeline) Live C2 predictions as input L2 error ↓ May worsen ← the key observation
The diagnostic logic: If C3 (pipeline) is worse than C3 (isolated), but C3 (isolated) is unchanged, the fault is upstream. C3 itself is fine. Cascade degradation has propagated from C1 through C2 into C3's input.
Section VI — Interpretation

What a Positive Result Confirms

If the experiment shows C3 pipeline degrades while C3 isolated stays the same, three things are confirmed simultaneously.

Confirmation 1
Cascade exists

Entangled enhancement propagates across two hops — C1 → C2 → C3 — in a real autonomous driving pipeline. Not theoretical. Measured.

Confirmation 2
Fault is upstream

C3 itself is not broken. The problem is the degraded input it receives. This validates the three-tier framework's ability to diagnose the source.

Confirmation 3
Framework works

The Tier 2 isolation technique from Meeting 2 correctly separates C3's own performance from upstream-induced degradation.

This single result justifies all subsequent work — the 8-state SMP, the cascade coverage factor ccas, and the full policy analysis. Everything else builds from this answer.
